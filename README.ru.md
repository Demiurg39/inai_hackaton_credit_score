# FinGuard — Telegram-бот финансовый охранник

> Строгий, но заботливый родитель, который **одобряет ✅ или блокирует ⛔** каждую покупку на основе твоего дневного лимита трат.

---

## Как это работает

### Основная идея

При настройке бота ты вводишь: текущий баланс, резерв (деньги которые не трогаем), и дату следующей зарплаты. Бот вычисляет **дневной лимит** — сумму, которую ты можешь безопасно тратить каждый день чтобы не остаться без денег до зарплаты.

Когда спрашиваешь про покупку, бот:
1. Сравнивает её с дневным лимитом
2. Запускает **симуляцию Монте-Карло** (2000 сценариев) — оценивает вероятность дотянуть до зарплаты
3. Применяет **нечёткую логику (fuzzy score)** — штрафует за перерасход
4. Выносит вердикт: **одобрено** (✅) или **отклонено** (⛔) с объяснением в тёплом родительском стиле

## Алгоритмы

### 1. Расчёт дневного лимита

```
available = баланс - резерв
дневной_лимит = available / дней_до_зарплаты
```

Если `available = 15000₽` и `дней до зарплаты = 10`, дневной лимит = `1500₽`.

### 2. Процент превышения (Overshoot)

```
overshoot_pct = (сумма_покупки - дневной_лимит) / дневной_лимит × 100
```

Покупка `3000₽` при лимите `1500₽` = `100% overshoot`.

### 3. Fuzzy Score (0.0 → 1.0)

Мягкий скоринг, который штрафует перерасход:

| Overshoot | Fuzzy Score |
|-----------|-------------|
| 0% | 1.0 (идеально) |
| 40% | 0.65 |
| 80% | 0.20 |
| >80% | 0.0–0.15 |

Формула:
```
overshoot ≤ 0%    → score = 1.0
overshoot ≤ 40%   → score = 1.0 - (overshoot/40) × 0.35
overshoot ≤ 80%   → score = 0.65 - ((overshoot-40)/40) × 0.45
overshoot > 80%   → score = max(0.15 - ((overshoot-80)/200) × 0.15, 0.0)
```

### 4. Симуляция Монте-Карло

Оценивает **вероятность выживания** — шанс не остаться без денег до зарплаты.

1. Берём `remaining_available` после покупки
2. Делим на оставшиеся дни → `mean_daily_after`
3. Используем персональное `std_daily_spend` (изученное из истории транзакций) как разброс
4. Запускаем 2000 симуляций: каждый день случайная трата из `N(mean, std)`
5. Считаем сколько симуляций дожили до конца периода без нуля
6. `survival_probability = surviving_sims / 2000`

Симуляция выполняется в thread pool через `asyncio.to_thread()` — не блокирует событийный цикл.

### 5. Масштабирование по толерантности к риску

У каждого пользователя есть `risk_tolerance` (0.0 = строгий, 1.0 = лояльный). Пороги сдвигаются:

```
fuzzy_threshold    = 0.52 - (risk_tol - 0.5) × 0.4    → 0.32 (строгий) … 0.72 (лояльный)
survival_threshold = 0.65 - (risk_tol - 0.5) × 0.5    → 0.40 (строгий) … 0.90 (лояльный)
```

Итоговое решение: `fuzzy > fuzzy_threshold AND survival > survival_threshold`

### 6. Полоска здоровья (Health Bar)

Отвечает на вопрос: "Мы идём по плану?"

```
идеальный_остаток = period_available × (1 - дней_прошло / всего_дней)
health = available / идеальный_остаток
```

Пример: бюджет 30000₽ на 30 дней. День 15 → идеальный остаток 15000₽. Если реально 10000₽, health = 67%.

### 7. Рекомендация резерва (Фаза 3)

Когда бот обнаруживает регулярные расходы (подписки, аренда и т.д.):

```
covered_recurring = сумма всех регулярных расходов
buffer = max(avg_daily_spend × 3, 500₽)   # минимум 500₽ буфер
recommended_reserve = covered_recurring + buffer
```

Если регулярные расходы не обнаружены: `recommended = баланс × 0.15` (фолбэк 15%).

## Возможности

### Оценка покупки
- Свободный текст: `300 обед`, `хочу купить кофе за 250`, `1200` (только сумма)
- Предсказание категории через Triton (например, "стрижка 800" → "Красота")
- Вердикт: одобрение с обновлением баланса ИЛИ блокировка с объяснением

### Движок решений (Фаза 2)
Правиловый caring-parent reasoning — **без LLM**:
- `block` — когда survival < 40% или баланс упадёт ниже 50% резерва
- `warn` — когда survival < 65% или overshoot > 50%
- `ok` — во всех остальных случаях

### Режим «Что если?» (Playground)
Напиши `🎮 Что если?` — посмотри вердикт по любой покупке **без сохранения транзакции и без изменения баланса**.

### Ежедневные уведомления (Фаза 3)
Опциональные утренние напоминания в удобное время (по умолчанию 10:00):
- Текущий баланс + дневной лимит
- Полоска финансового здоровья
- Мотивирующее сообщение

### Проактивные алерты (Фаза 3)
Бот автоматически предупреждает когда:
- Предсказанная дата конца денег раньше дня зарплаты
- Крупный регулярный платёж (аренда и т.д.) ожидается в течение 7 дней и может превысить бюджет

### Персонализированная статистика (Фаза 1)
Бот учится из истории транзакций:
- `avg_daily_spend` / `std_daily_spend` — реальный паттерн трат
- `spend_velocity` — как меняется скорость трат со временем
- `risk_tolerance` — настраивается в `/settings`
- Обнаружение регулярных расходов (сумма + интервал)

## Структура проекта

```
finguard/
├── bot.py                        # Точка входа, регистрация роутеров
├── config.py                     # BOT_TOKEN из .env
├── database/
│   ├── db.py                    # SQLite инициализация, миграции
│   └── models.py                # CRUD: пользователи, транзакции, статистика
├── handlers/
│   ├── start.py                 # /start — 3-шаговая FSM настройка
│   ├── purchase.py              # Оценка покупок + режим «Что если?»
│   ├── status.py                # /status — health bar + прогноз + рекомендация резерва
│   ├── settings.py              # /settings — настройки + история
│   └── notifications.py         # /notify — ежедневные уведомления
├── services/
│   ├── calculator_advanced.py   # Ядро: fuzzy + Monte Carlo
│   ├── reason_engine.py         # Правиловый движок вердиктов
│   ├── explainer.py            # Форматтер сообщений в стиле заботливого родителя
│   ├── reserve_advisor.py       # Рекомендация размера резерва
│   ├── proactive_alerts.py      # Проактивные алерты
│   └── triton.py                # Предсказание категории через Triton
├── middlewares/
│   ├── onboarding_check.py      # Перенаправление ненастроенных пользователей на /start
│   └── notification.py         # Ежедневные уведомления при каждом poll
└── states/
    └── fsm.py                   # FSM группы состояний
```

## Запуск

```bash
# Установить зависимости
pip install -r requirements.txt

# Настроить токен
cp .env.example .env
# Вписать BOT_TOKEN=<токен от BotFather>

# Запустить
python bot.py
```

База данных SQLite (`finguard.db`) создаётся автоматически при первом запуске.

## Переменные окружения

| Переменная | Обязательно | Описание |
|---|---|---|
| `BOT_TOKEN` | ✅ | Токен Telegram-бота от BotFather |
| `DB_PATH` | ❌ | Путь к SQLite (по умолчанию: `finguard.db`) |
| `TRITON_URL` | ❌ | Triton Inference Server |
| `LLM_URL` | ❌ | LLM endpoint (не используется — Фаза 2 работает без него) |
